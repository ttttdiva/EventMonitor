# 開発ガイドライン

EventMonitorの開発フローと運用上の注意をまとめたドキュメントです。詳細な仕様はREADMEや各種docsに委譲し、ここでは開発者として守るべきルールと参照先を示します。

## 基本プロセス
1. 仕様理解 → 計画共有（Planツールが必要なら使用） → 実装 → テスト → 報告の順番を必ず守る。
2. 実装完了＝テスト完了。正常系と異常系の少なくとも1ケースずつを意識し、実行できない場合は理由を報告する。
3. 途中経過や疑問がある場合は早めに確認し、独断で仕様を補わない。

## 実装前チェック
- `.env`や`config.yaml`のキー名を既存コードと突き合わせてから変更する。
- 参照するAPI/外部ツールのドキュメントを確認し、パラメータの前提を揃える。
- 必要なファイルパスや権限を事前に確認し、欠けている場合は必ず報告する。

## 作業管理
- 複雑な作業ではPlanツールで手順を宣言し、完了ごとに更新する。
- TodoWrite等に頼らずとも、必要なToDoを明文化してから着手する。
- エラーが発生したら場当たり的に直さず、まず原因を分析する。
- `git push`を求められた場合は、実際に必要な変更がコミットされているか確認し、`git add`→`git commit`→`git push`を一連で行う。
- コミットメッセージは「どこを」「なぜ」変えたか分かるように具体的に書き、機能追加とドキュメント更新など複数要素があれば要約を入れる。

## 技術上の注意
- DB操作はSQLAlchemy経由。システム同梱の`sqlite3` CLIは使用禁止。Pythonからアクセスする場合は`pysqlite3-binary`を使う（`src/database.py`参照）。
- gallery-dlは`cookies/x.com_cookies_*.txt`をローテーションして使用。twscrapeは`.env`の`TWITTER_ACCOUNT_*_TOKEN`/`CT0`を使用する。
- レート制限回避のため、複数アカウントのローテーションとリツイート/リポスト除外ロジックを壊さない。
- Hydrus連携のタグ仕様やHuggingFaceバックアップ手順は専用ドキュメントに合わせる。

## テスト・チェック
- テスト体系と推奨コマンドは`docs/testing_strategy.md`に記載。lint/セキュリティチェックは`python run_tests.py --lint` / `--security`。
- テストを省略する場合は省略理由＆代替の確認方法を報告する。
- 代表的な動作テストの流れは`docs/system_flow.md`や`docs/system_design.md`内のシーケンス図を参照。
