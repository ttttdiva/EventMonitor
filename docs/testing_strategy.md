# EventMonitor テスト戦略（2025年12月時点）

## 現状まとめ
- 歴史的に多数のpytestスクリプトや手動チェック用ツールが混在していましたが保守されておらず、最新コードとは乖離していました。
- メンテ不能なテストは削除し、**最小限のユニットテスト群**から再構築しています。現在はデータベースフィルタリングの回帰を守る`tests/unit/test_database_filters.py`のみが提供されています。
- 今後のプルリクでは、ここを起点に優先領域（データベース、Hydrus連携、Twitterモニタ実行フロー）へ順次テストを追加していきます。

## ディレクトリ構造
```
tests/
├── conftest.py                 # 共通の環境変数セットアップとパス調整のみ
└── unit/
    └── test_database_filters.py  # DatabaseManagerのフィルタ処理テスト
```

## 実行方法
```bash
pytest tests/unit -v
```
- 追加でテストを作成した場合も `pytest tests` で一括実行する方針です。
- 依存サービス（Twitter APIやHydrusなど）はモック化し、CI環境でも動作する形にしてください。

## 既知のギャップ
1. **EventDetector / DiscordNotifier** などLLM・Webhook依存コードのユニットテストが未整備。
2. **TwitterMonitor～BackupManager** の統合テストが存在しないため、将来はモックベースのフローテストを新設する必要があります。
3. 旧テストスイートで参照していた`tests/fixtures/`や大量の補助スクリプトは削除済みです。必要になれば各テストでローカルフィクスチャを定義してください。

## 新しいテストを追加する際の指針
- モジュール単位の責務を明確にし、外部I/Oはフィクスチャまたはモックに切り出す。
- テスト名は `test_<対象>_<条件>_<期待>` の形式で3語以上になるようにし、失敗時に原因がわかる説明を含める。
- CI負荷を抑えるため、ネットワーク呼び出しや外部バイナリ実行を伴うテストは `pytest.mark.slow` を定義した上でデフォルトから除外する。
- 依存するテストデータが複数ケースで再利用できる場合のみ `tests/fixtures/` を再作成する。それ以外は各テストでインライン定義する。

## ロードマップ（提案）
